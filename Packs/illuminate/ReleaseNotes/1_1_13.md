
#### Integrations

##### Analyst1

- **Breaking Change**: Updated XSOAR Verdict calculation logic to use Analyst1 Risk Score instead of the benign field.
  - **Previous behavior**: Verdict was set to Malicious (3) when benign=false, and Benign (1) when benign=true. Since Analyst1 defaults benign=false for all indicators, this caused false positives.
- Added 6 new configuration parameters to map Analyst1 Risk Scores to XSOAR Verdicts:
  - Risk Score Mapping: Lowest (default: Benign)
  - Risk Score Mapping: Low (default: Unknown)
  - Risk Score Mapping: Moderate (default: Suspicious)
  - Risk Score Mapping: High (default: Suspicious)
  - Risk Score Mapping: Critical (default: Malicious)
  - Risk Score Mapping: Unknown (default: Unknown)
- Updated all 9 enrichment commands (domain, email, ip, file, url, analyst1-enrich-string, analyst1-enrich-ipv6, analyst1-enrich-mutex, analyst1-enrich-http-request) to:
  - Extract and return the Analyst1 Risk Score in context output
  - Calculate XSOAR Verdict using the new risk score-based logic
  - Automatically tag indicators with "Analyst1: Indicator"
- Updated batch-check commands (analyst1-batch-check, analyst1-batch-check-post) to:
  - Set XSOAR Verdict and DBotScore based on entity types and risk scores
  - Verdict is set to Benign (1) when `benign=true` or when `entity.key` is ASSET, IN_SYSTEM_RANGE, IN_HOME_RANGE, IN_PRIVATE_RANGE, IGNORED_INDICATOR, or IGNORED_ASSET
  - Verdict is set based on risk score mapping when `entity.key=INDICATOR`
  - Verdict is set to Unknown (0) when `indicatorRiskScore` is null
  - Automatically manage XSOAR tags based on entity types (e.g., "Analyst1: Asset", "Analyst1: In Home Range")
  - Support multiple tags per indicator when multiple entity types are returned
  - Dynamically add/remove tags to reflect current Analyst1 classification
