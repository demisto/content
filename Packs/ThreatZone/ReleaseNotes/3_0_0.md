#### Integrations

##### ThreatZone

- Migrated the integration to version **1.1.1** of the official `threatzone` Python SDK. API traffic, authentication, retry logic, and response parsing flow through the SDK's synchronous client.
- Targets Threat.Zone platform **v3.2.0+** and SDK **1.1.1**.
- Preserved every existing command (`tz-sandbox-upload-sample`, `tz-static-upload-sample`, `tz-cdr-upload-sample`, `tz-url-analysis`, `tz-get-result`, `tz-get-indicator-result`, `tz-get-ioc-result`, `tz-get-yara-result`, `tz-get-artifact-result`, `tz-get-config-result`, `tz-get-sanitized`, `tz-download-html-report`, `tz-check-limits`) and the top-level context paths (`ThreatZone.Submission.*`, `ThreatZone.Analysis.*`, `ThreatZone.IOC.*`, `ThreatZone.Limits.*`, `ThreatZone.Plan.*`, `ThreatZone.Metadata.*`) so existing playbooks continue to function.
- Legacy `ThreatZone.Analysis.LEVEL` and `ThreatZone.Analysis.STATUS` integer codes are preserved by mapping the SDK's `ThreatLevel` and `ReportStatusValue` enums back to the historical integers (`0`–`3` for level, `0`–`5` for status). Playbook branches on these numeric values keep working.
- SDK-typed exceptions (`AuthenticationError`, `PermissionDeniedError`, `RateLimitError`, `ReportUnavailableError`, etc.) are translated into structured error messages in the War Room rather than raw tracebacks.
- Improved XSOAR 6.x runtime compatibility by avoiding module directives that conflict with runtime-injected common code.
- Updated sandbox presets to the Threat.Zone v3.2 environment identifiers (`linux-ubuntu_2204`, `macos-ventura`, and `android9`) and stopped forwarding the unsupported legacy `raw_logs` dynamic-report metafield.
- Sandbox defaults are now loaded from ThreatZone's metafield configuration API. Active and accessible API defaults, such as `snapshot=true`, are applied unless explicitly overridden by an XSOAR command argument or `analyze_config`.
- Added commands for every public ThreatZone Python SDK 1.1.1 operation, including configuration discovery, open-in-browser and submission queries, detailed report sections, bounded telemetry, and file/media downloads.
- Made the server URL mandatory without a cloud fallback and added normalization for instance-root and `/public-api` URLs.
- Added sandbox execution configurations, URL safe-browsing, and indicator, IOC, and matched-YARA filters.
- Fixed report retrieval to return every page of filtered indicators, IoCs, and matched YARA rules, including the legacy `ThreatZone.IOC` output.
- Restored streaming for SDK downloads to avoid buffering large files in integration memory, and added generated-YARA readiness polling and safe media filename validation.
- Preserved original War Room filenames for sandbox, static, and CDR uploads.
- Kept the legacy `modules` argument as a validated compatibility no-op because ThreatZone v3.2 does not support per-submission module selection.
- Aligned the supplied HTTP transport with the SDK's 60-second timeout and redirect behavior, and preserved the legacy `urlAnalysis` report type label.
- Updated to the immutable production image `demisto/threatzone-sdk:1.1.1.11256202` and removed the obsolete `os` argument from `tz-get-behaviours`, matching the corrected SDK and public API contract.

##### Note

Existing playbook YAMLs and external context references do not need changes. The internal shape of `ThreatZone.Submission` reflects the new public API schema (camelCase pydantic dump) because the upstream `/public-api/submissions/{uuid}` endpoint replaced the legacy `/get/submission/{uuid}` endpoint in Threat.Zone v3.2.0. Customers running against pre-v3.2.0 instances must pin to ThreatZone content pack `2.0.1` until they upgrade the platform.
