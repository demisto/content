
#### Integrations

##### Doppel

- Fixed an issue in **fetch-incidents** that could create many duplicate incidents from a single Doppel alert. Alerts are now de-duplicated by their Doppel alert id both within a run and across runs.
- Fetch now processes alerts page by page with a bounded number of pages per run, so a large backlog is drained safely over consecutive runs without timing out or looping.
- Alerts are now fetched using the maximum page size (200) supported by the Doppel API to reduce the number of requests.
- Incidents are now named using the external Doppel alert id (for example, **Doppel Alert TET-1953421**) and set **dbotMirrorId** explicitly for reliable mirroring and de-duplication.
- Blank or invalid **Fetch Timeout** and **Maximum incidents per fetch** values are now handled gracefully.

#### Scripts

##### New: DoppelDedupeIncidents

- Added the **DoppelDedupeIncidents** automation to consolidate duplicate Doppel incidents created before the fetch fix. It groups incidents by Doppel alert id, keeps a single canonical open incident (the oldest open incident with an owner, otherwise the oldest open incident), and closes (default, reversible) or optionally deletes the remaining open duplicates. Already-closed incidents are never touched, and runs are idempotent so a large backlog can be drained in batches.
