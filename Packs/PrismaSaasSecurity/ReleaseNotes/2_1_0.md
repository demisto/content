
#### Integrations

##### SaaS Security Event Collector

- Improved fetch throughput by draining the SaaS Security queue with concurrent requests, so the collector keeps pace with high upstream event rates and ingestion lag stays low.
- Fixed an issue where concurrent fetches could intermittently fail with JSON decode errors and cause a silent ingestion stall. Each concurrent request now uses its own connection (session), avoiding the shared-connection race that corrupted responses.
- Fixed an issue where a previously stashed batch of events that failed to send could be retried indefinitely. Events are now sent to Cortex XSIAM in smaller chunks and only the unsent remainder is retried, so a stuck backlog drains and recovers on its own.
- The effective *maximum number of iterations to retrieve events* is now automatically raised to at least 900, giving each fetch cycle enough headroom to keep draining a sustained high-volume backlog for the full safe time budget (each cycle still stops within that budget before the execution timeout). An instance configured with a lower legacy value is not throughput-limited.
- Fixed an issue where a fetch could be terminated by the execution timeout while draining a large backlog, losing in-progress state. Each fetch now stops within a safe time budget, persists its progress, and immediately continues on the next cycle.
- Added a *Number of concurrent fetch requests* parameter to tune drain concurrency for high event rates.
- Added a high-visibility log when the event queue does not fully drain for several consecutive fetch cycles, so sustained ingestion backlog is observable.
- Added diagnostic logging that captures the actual response body returned by Cortex XSIAM when a send fails, to distinguish an empty or truncated ingestion response from a payload problem.
- Added a *Treat an empty Cortex XSIAM response as delivered* parameter (enabled by default). When Cortex XSIAM returns a success (200) with an empty body and no acknowledgement, the events are treated as delivered instead of being retried indefinitely; truncated or otherwise invalid responses are still retried.
- Added a lock around server calls during concurrent fetch so parallel worker threads cannot interfere with the integration's communication with the server.
